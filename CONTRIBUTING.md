# Правила командной разработки Evidence-Locker

> [!IMPORTANT]
> **Локальная разработка и тестирование**
> Запуск самого приложения (FastAPI) и базы данных происходит **строго в Docker-контейнере** (`docker-compose up`). Однако, для удобной работы в IDE (автокомплит) и быстрого локального запуска инструментов разработчика (линтеры, тесты) настоятельно рекомендуется установить `uv` на вашу хост-машину.

## 1. Workflow (Git Flow)
Использование Feature-веток:
- Создавайте ветки от `main`. Название ветки должно отражать суть задачи (например, `feature/1-models-setup`, `fix/login-bug`).
- Вся работа ведется через Pull Requests.
- PR должен быть проверен хотя бы одним участником команды (ревью) перед вливанием в `main`.

## 2. Типы коммитов (Conventional Commits)

Строгое использование [Conventional Commits](https://www.conventionalcommits.org/):
* **feat**: Добавление нового функционала.
* **fix**: Исправление ошибки (бага).
* **docs**: Изменения только в документации (например, README.md).
* **style**: Изменения форматирования, не влияющие на логику кода (отступы, пробелы, линтер).
* **refactor**: Изменение кода, которое не добавляет новый функционал и не исправляет баг.
* **perf**: Изменения, направленные на улучшение производительности.
* **test**: Добавление новых или исправление существующих тестов.
* **build**: Изменения, влияющие на систему сборки или внешние зависимости (requirements.txt, Docker).
* **ci**: Изменения в конфигурации CI/CD пайплайнов (например, GitHub Actions).
* **chore**: Рутинные задачи и обслуживание, не влияющие на исходный код приложения (настройка окружения, скаффолдинг).

## 3. Ежедневная отчетность
Напоминание об обязательном ведении `.md` файлов (заметок) по задачам SMART со ссылками на PR. Это необходимо для успешной защиты практики.

## 4. Настройка окружения (Переменные и Токены)

> [!WARNING]
> Если вы попытаетесь запустить приложение без файла `.env`, Docker выдаст ошибку о том, что файл не найден, или приложение упадет из-за отсутствия токенов аутентификации.

Перед первым запуском приложения обязательно создайте локальный файл с переменными окружения. Для этого скопируйте файл с примерами:

```bash
cp .env.example .env
```

Файл `.env.example` содержит необходимые для локальной работы (и прохождения демо-скрипта) фиктивные токены:
- `TEACHER_TOKEN`
- `COLLECTOR_TOKEN`
- `STUDENT_TOKEN`

Никогда не добавляйте ваш локальный файл `.env` с реальными секретами в коммит! (Он уже добавлен в `.gitignore`).

## 5. Запуск контейнера и работа с `uv`
Вся работа с приложением и зависимостями ведется через Docker:
- **Сборка и запуск:** `docker-compose up --build`
- **Добавление/обновление зависимостей:** 
  Так как файл `pyproject.toml` копируется в контейнер при сборке (а не пробрасывается как volume), **изменять его нужно только локально на вашей хост-машине**.
  1. **Вручную:** Откройте файл `pyproject.toml` в вашем локальном редакторе кода и впишите новую зависимость в секцию `dependencies = [...]`.
  2. **С помощью `uv`:** Если у вас локально установлена утилита `uv`, выполните в терминале на хост-машине:
     `uv add <имя_пакета>`
     *(Никогда не делайте это через `docker-compose exec`, иначе при пересборке зависимость исчезнет!)*
  3. После локального изменения `pyproject.toml` обязательно пересоберите контейнер:
     `docker-compose up --build -d`
- **Просмотр логов:** `docker logs <container_name>` (логи о событиях пишутся в stdout)

## 6. Установка `uv` на хост-машину

Если вы хотите запускать dev-инструменты локально и использовать `uv` для быстрого добавления пакетов:

**macOS / Linux:**
```bash
curl -LsSf https://astral.sh/uv/install.sh | sh
```

**Windows:**
```powershell
powershell -ExecutionPolicy ByPass -c "irm https://astral.sh/uv/install.ps1 | iex"
```

После установки вы можете запускать утилиты разработчика прямо из терминала. Команда `uv run` автоматически создаст и обновит виртуальное окружение `.venv` в папке проекта:
- `uv run ruff check .` (запуск линтера)
- `uv run pytest` (запуск тестов)
- `uv sync` (команда синхронизации локального окружения). **Обратите внимание:** эта команда установит локально *все* пакеты (и основные, и dev). Это необходимо, чтобы ваша IDE видела библиотеки (FastAPI, SQLAlchemy и др.) и корректно работала с автокомплитом и проверкой типов, даже если сам код будет запускаться в Docker. *Совет: при добавлении новых локальных утилит используйте `uv add --dev <имя_пакета>`, чтобы они не попали в основные зависимости.*

## 7. Линтинг и форматирование
Требуется использовать `ruff` для проверки и форматирования кода перед коммитом.
- **Локально (через uv):** 
  `uv run ruff check .`
  `uv run ruff format .`
- **Через Docker (если uv не установлен):**
  `docker-compose exec api ruff check .`
  `docker-compose exec api ruff format .`
